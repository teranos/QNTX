# config — QNTX Core Configuration

Configuration package for QNTX. Loads and validates settings from a five-source cascade.

## What belongs in `config`

- Infrastructure: database, server, async jobs (Pulse)
- Core interfaces: code review, attestation queries (ax)
- Generic capabilities: local LLM inference, HTTP rate limiting

## What does NOT belong

- Domain entities: contacts, organizations, roles, candidates
- Business logic: scoring algorithms, matching weights
- External integrations: social media APIs, cloud services
- Application-specific features: recruitment workflows, event management

Domain-agnostic infrastructure. Applications extend it for their needs.

## Configuration Precedence

Loaded in order (lowest to highest priority):

1. System: `/etc/qntx/config.toml` (or `am.toml`)
2. User: `~/.qntx/config.toml` (or `am.toml`)
3. User UI: `~/.qntx/am_from_ui.toml` (auto-generated by web UI)
4. Project: `./config.toml` (or `am.toml`) — searches parent directories
5. Environment variables: `QNTX_*` prefix (e.g. `QNTX_DATABASE_PATH`)

**File naming**: accepts both `am.toml` (legacy) and `config.toml`.

Sensitive values (API keys, tokens) should use environment variables.

See [Configuration System Architecture](../../docs/architecture/config-system.md) for rationale, introspection, and implementation.

## Configuration Sections

- **`database`** — SQLite storage path and settings
- **`server`** — web UI ports, CORS origins, log themes
- **`pulse`** — async job workers, scheduling, HTTP rate limiting
- **`code`** — code review system (GitHub integration)
- **`local_inference`** — local LLM support (llama.cpp plugin)
- **`ax`** — attestation query defaults

See `am.go` for complete type definitions.

## Extending in Applications

Applications extend `config.Config` for domain-specific needs through dual loading:

1. Load QNTX core config (`config.Load()`)
2. Load application domain config (custom loader)
3. Merge at runtime into unified config structure

Both can read from the same `config.toml` file — `config` extracts core keys, applications extract domain keys. Unknown keys are ignored for compatibility.

## Backward Compatibility

- Reads both `am.toml` (legacy) and `config.toml`
- Ignores unknown keys (domain fields won't break the loader)
- Same TOML structure, extracts core subset
